Eval Runner
eval/runner.py
asyncio · 3 backends concurrent per question · NaN on per-backend error · partial write after each question

The core evaluation engine sends each question to all three knowledge systems at once, scores each answer independently, and saves progress after every question so a crash doesn't lose work.

eval/backends.py
Vector RAG · Graph RAG · Cognee · each: async (question) → (context_str, summary)
This component knows about all three AI knowledge systems. It sends the same question to each and returns the context each one found, ready for scoring.

Testset
eval/testset.py · data/testset.json
20 Q&A pairs · atomic os.replace() write · schema-validated on load
A fixed set of 20 questions with known correct answers, generated from the Everstorm product PDFs. Every evaluation run uses the same questions, making results comparable over time.

eval/metrics.py · ragas.evaluate()
faithfulness · answer_relevancy · context_precision · context_recall · InstructorLLM + HuggingFace embeddings
The scoring engine objectively measures each answer across four dimensions: whether it sticks to the source, whether it answers the question, and whether the right information was retrieved.

BATCH EVALUATION FLOW
📂
1 · Load Testset
load_testset() → data/testset.json
validates schema · raises FileNotFoundError with docker exec hint if missing
The first step loads 20 pre-built questions and their correct answers. If the file is missing, the evaluation stops with clear instructions for how to regenerate it.
20 Q&A
🔁
2 · Sequential Loop
for item in testset: … asyncio.gather()
questions run one at a time · atomic tmp→replace partial write after each · stale partial cleared on next run
Questions are processed one at a time to avoid overloading the AI APIs. After each question, partial results are saved — if the run is interrupted, the partial file records what completed.
sequential
⚡
3 · Concurrent Backends
asyncio.gather(*tasks) · _eval_one_backend × 3
Vector RAG · Graph RAG · Cognee fire simultaneously · NaN + error string on per-backend failure
✓ Per-backend isolation: one fail ≠ all fail
For each question, all three knowledge systems are queried at the same time. If one fails, the others continue normally — that system simply shows a blank score instead of crashing the whole run.
concurrent
📐
4 · Score Each Answer
_run_ragas() → ragas.evaluate() · asyncio.to_thread()
claude-haiku-4-5-20251001 judge · max_tokens 4096 · BAAI/bge-small-en-v1.5 via HuggingFace HTTP
Each answer is scored by Claude acting as an impartial judge. It reads both the answer and the source material to measure accuracy, relevance, and whether the right content was retrieved.
4 metrics
📊
5 · Aggregate Results
df.groupby("backend")[METRIC_NAMES].mean().round(4)
mean per backend · returned as DataFrame · displayed in Gradio Dataframe component
Once all 20 questions are evaluated, scores are averaged per system. The result is a comparison table showing which knowledge system performs best across each of the four quality dimensions.
mean scores

🎛
1 · Event Handler
handle_single_eval() · app/handlers.py
question + optional ground_truth · yields gr.update(interactive=False, value="Evaluating…") first · generator pattern
When a user types a question and clicks Evaluate, the button immediately disables to prevent duplicate submissions. The handler then waits for all three systems to respond before showing results.
generator

2 · Concurrent Eval
run_single_eval() · asyncio.gather × 3
Vector RAG · Graph RAG · Cognee all fire simultaneously · NaN on per-backend error
All three knowledge systems receive the question at the same moment. Total wait time equals the slowest system — not the sum of all three — because they run in parallel.
concurrent
📊

Each system's answer is scored and shown side-by-side in three columns, letting you directly compare what each retrieved, what it said, and how each quality dimension scored it.
